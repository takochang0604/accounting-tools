# Delivery Quality

適用範圍：所有需要交付可使用成果的任務，包含前端頁面、工具、小型產品、CRUD 介面、表單流程、Firebase 工具、互動式頁面與多檔案整合功能。

## 核心原則
- 交付內容不只要有功能，還應兼顧美觀、可讀性、使用體驗、流程完整性與基本可落地性。
- 不可只滿足字面需求而忽略一般使用者實際會遇到的操作情境。
- 不可只完成局部功能、靜態畫面或半套流程就視為完整交付。
- 不可將明顯未完成、未檢查或尚未串起來的內容包裝成已可使用成果。

## 介面與視覺要求
- 前端介面應優先追求乾淨、清楚、現代、具基本商業工具感，不可做得像工程測試頁或功能拼裝頁。
- 畫面應有明確視覺層次，包含標題、區塊、主要資訊、次要資訊、操作區與提示訊息。
- 不可讓整個畫面資訊過度擁擠，也不可因過度極簡而缺乏必要資訊。
- 主要按鈕、次要按鈕、危險操作按鈕應有清楚區分，不可全部長得一樣。
- 重要資訊如狀態、金額、未完成項目、到期日、錯誤提示、成功提示等，應有足夠醒目的呈現方式。
- 若未指定風格，預設應採簡潔、易讀、現代、一般使用者容易接受的設計方向。

## 使用體驗要求
- 應從一般使用者角度思考畫面與流程，而不是只從資料結構或工程角度出發。
- 使用者應能快速理解畫面用途、主要操作與下一步。
- 常用操作應明顯且容易找到，不可埋得太深。
- 操作流程應自然順手，不應讓使用者需要反覆猜測按哪裡、存到哪裡、下一步是什麼。
- 新增、編輯、刪除、查看詳情、儲存、取消、返回、切換狀態等常見動作應有清楚入口與合理回饋。
- 編輯流程應帶入既有資料，不可要求使用者每次重填全部欄位。
- 刪除、清空、重設、登出等高風險操作應提供確認機制。

## 表單體驗要求
- 表單欄位順序應符合一般人理解流程，而不是依資料欄位隨意排列。
- 必填欄位、選填欄位、預設值、欄位說明、錯誤提示應清楚。
- 重要欄位、常用欄位、會影響後續流程的欄位應優先出現。
- 欄位過多時，應進行區塊分組、分段或分頁，而不是全部堆成一長串。
- 送出前應有基本驗證；送出失敗時應有清楚錯誤提示，不可默默失敗。
- 儲存後應提供清楚回饋，必要時同步更新畫面或返回適當位置。

## 列表、卡片與資料呈現要求
- 列表、卡片、表格應優先顯示最常被查看與比較的重要資訊。
- 不可資訊過少到必須不停點進詳情，也不可資訊過多到難以掃讀。
- 相同類型的資料呈現方式應一致，不可每區塊邏輯不同。
- 若資料量可能增加，應主動考慮合理的搜尋、篩選、排序、分類、展開/收合、分頁或懶載入。
- 詳情頁應補足列表未展開的內容，並保有清楚的返回、編輯、刪除等入口。

## 常見功能主動補齊要求
- 對於常見工具類型，不可每次都只做最小字面需求，應主動補齊一般使用者預期會有的基本功能。
- 若需求屬於 CRUD、列表管理、記帳工具、管理頁、表單工具、訂閱管理工具、收藏工具、待辦工具、會員工具等，應主動評估是否需要以下常見能力：
  - 搜尋
  - 篩選
  - 排序
  - 分類
  - 分頁或懶載入
  - 空狀態提示
  - loading 狀態
  - success / error 提示
  - 刪除確認
  - 表單驗證
  - 編輯帶入原值
  - 預設值或快速選項
  - 列表與詳情之間的合理切換
- 若判斷某些常見功能在當前版本不需要加入，也應基於合理產品判斷，而不是單純因為使用者沒提就忽略。

## 狀態與回饋要求
- 應主動處理 loading、empty state、success、error、disabled、saving / submitting 等常見狀態。
- 不可讓使用者按下按鈕後沒有回應，或完成操作後不知道是否成功。
- 無資料時不可只顯示空白畫面，應有可理解的空狀態說明與下一步引導。
- 發生錯誤時不可只在 console 報錯，應提供使用者可理解的提示。
- 涉及遠端資料操作時，應顯示基本載入或提交中狀態，避免誤判系統無反應。

## 響應式與裝置適配要求
- 不要求所有需求都做到完整多裝置最佳化，但至少要確保常見桌面與手機寬度下基本可用。
- 不可明顯爆版、截斷、按鈕過小、欄位難以輸入、彈窗超出視窗或主要操作被遮住。
- 小螢幕下主要操作仍應可找到與點擊。

## 功能完整性要求
- 交付前應對照原始需求，檢查功能是否完整覆蓋。
- 不可只做畫面而未接上功能。
- 不可只做新增而漏掉查詢、編輯、刪除、狀態切換等合理環節。
- 不可有按鈕存在但未綁事件、函式存在但沒有入口、資料欄位顯示但未保存、畫面存在但流程斷裂。
- 若需求中明確提到某頁面、某區塊、某功能、某流程，不可遺漏。
- 不可只完成資料結構說明或局部程式碼，卻將整體視為完成交付。
- 若需求涉及多步驟流程，至少應確認主要流程有首尾相連，而不是零碎片段。

## 交付前自我檢查要求
- 交付前應至少進行一輪自我檢查，不可寫完直接交。
- 應檢查檔名、import / export、script 路徑、CSS / JS 載入、DOM selector、初始化流程、事件綁定是否一致。
- 應推演主要流程是否連得起來，例如：
  - 進入頁面
  - 新增
  - 編輯
  - 刪除
  - 儲存
  - 列表更新
  - 詳情切換
  - 登入 / 登出
- 若涉及 CRUD 或遠端資料，應檢查新增後是否顯示、更新後是否同步、刪除後是否移除、空資料時是否正常。
- 若涉及表單，應檢查必填驗證、錯誤提示、送出回饋、編輯預填與提交結果。
- 若無法實際執行驗證，應至少進行靜態檢查與流程推演，並誠實標示哪些部分尚未實測。

## 誠實交付要求
- 若有未完成項目、限制、假設、外部依賴、待補設定或尚未驗證部分，必須清楚標示。
- 不可將未串接、未驗證、未完成的內容說成已可直接使用。
- 不可用模糊話術掩蓋功能缺口或驗證不足。
- 交付時應盡量說明：
  - 已完成項目
  - 尚未完成項目
  - 需補設定部分
  - 已知限制
  - 後續建議檢查點

## 最終要求
- 交付成果應盡量像一個可用版本，而不是只像功能草稿。
- 在不過度設計的前提下，應優先交付具有基本產品感、操作合理、功能完整、流程連得起來的結果。

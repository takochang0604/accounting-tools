# Coding Baseline

適用範圍：所有程式撰寫、修改、重構、除錯、補完任務。

## 硬性規範
- 帳號密碼驗證邏輯不可放在前端。
- 權限控管不可只依賴前端判斷、前端隱藏按鈕或前端顯示控制。
- 不可將 API key、secret、token、service account 憑證、資料庫密碼或其他敏感資訊直接寫在前端程式碼中。
- 涉及登入、權限、敏感資料、管理操作時，真正的安全判斷必須放在後端、受控服務或安全規則層。
- 不可為了快速完成而提供明顯不安全的實作方式。
- 不可將明顯未完成、無法使用或僅完成部分流程的內容包裝為完整交付。
- 不可只完成靜態畫面就視為功能完成。
- 不可忽略需求中已明確列出的功能項目，導致交付內容東漏西漏。

## 預設方向
- 資料存取類需求，預設優先考慮 Firebase 架構。
- 涉及身份識別時，優先考慮 Firebase Authentication。
- 涉及資料讀寫時，應同時考慮 Firestore / Storage Security Rules。
- 涉及敏感操作、管理權限、第三方密鑰保護或不適合直接由前端處理的邏輯時，優先考慮 Cloud Functions 或其他後端方案。
- 前端介面預設應優先追求清楚、易懂、易操作，而不是只追求功能堆疊。
- 若需求涉及表單、列表、CRUD、設定頁、詳情頁或彈窗，應同時考慮基本 UI/UX 可用性與操作流程合理性。
- 交付前應先自行檢查主要流程、功能完整性、檔案引用與事件綁定是否合理。

## 介面與體驗基本要求
- 畫面應有清楚的主題、主要操作區與次要資訊區，不可資訊過度堆疊。
- 主要按鈕與次要按鈕應有清楚區別，避免使用者誤操作。
- 表單欄位順序應符合一般人理解流程，不可隨意排列。
- 新增、編輯、刪除、儲存、取消、返回等常見操作應有清楚入口。
- 刪除、清空、重設等危險操作應有確認機制。
- 操作後應有基本回饋，例如成功、失敗、載入中、空資料等狀態提示。
- 無資料時不可只顯示空白畫面，應提供可理解的空狀態說明。
- 小螢幕下至少要維持基本可用，不可明顯爆版或無法操作。

## 交付前檢查與完整性要求
- 交付前應對照原始需求，檢查功能是否完整覆蓋。
- 應檢查是否存在只做了畫面、未接功能的情況。
- 應檢查是否存在按鈕未綁事件、函式未被呼叫、初始化未執行、檔案未正確引用等問題。
- 應至少推演主要流程是否連得起來，例如進入頁面、新增、編輯、刪除、儲存、列表更新、登入登出等。
- 應確認新增、編輯、刪除、查詢等常見 CRUD 流程未遺漏。
- 應確認必填欄位、基本驗證、錯誤提示與操作回饋未遺漏。
- 若無法實際執行驗證，應明確說明哪些部分已靜態檢查、哪些尚未實測、哪些仍需補設定後驗證。
- 若有未完成項目、限制或已知風險，必須誠實標示，不可隱藏或模糊帶過。

## 彈性原則
- 若有比 Firebase 更合理的方案，可以提出替代做法，但必須說明原因、風險與取捨。
- 可依需求複雜度調整架構，不需為了形式一致而過度設計。
- 在不違反安全底線的前提下，可提出更好的實作方式。
- 在不降低基本可用性與完整性的前提下，可依情境調整介面形式、流程安排與檔案拆分方式。
